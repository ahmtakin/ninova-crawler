# Production Deployment Recommendations

For a production deployment, the first priority would be separating the crawler workers from the API/search layer into independently scalable services. The crawler workers should be stateless processes that pull URLs from a distributed queue (e.g., Amazon SQS or a Redis Cluster–backed BullMQ setup), enabling horizontal scaling across multiple nodes partitioned by domain to respect per-host rate limits and politeness policies. The search layer should sit behind a load balancer with read replicas of the index store, ideally migrating the inverted index from MongoDB to a purpose-built search engine like Elasticsearch or Meilisearch that provides TF-IDF scoring, fuzzy matching, and sub-second query latency at scale. MongoDB would remain the system of record for crawl metadata and page content, deployed as a replica set with proper write concern and read preferences. Redis should be deployed as a cluster for the URL frontier and deduplication, with bloom filters replacing exact sets to reduce memory consumption at billions of URLs.

On the operational side, the system needs comprehensive monitoring and observability: structured logging shipped to a centralized platform (e.g., Datadog or Grafana/Loki), metrics dashboards tracking crawl throughput, error rates, queue depth trends, and search latency percentiles, plus alerting on back pressure events and worker failures. robots.txt compliance must be enforced with a per-domain cache and a configurable crawl-delay floor. A DNS resolution cache and connection pooling layer (with keep-alive) would significantly reduce per-request overhead. For data integrity, the crawl queue should use exactly-once processing semantics with dead-letter queues for poison URLs, and all state transitions should be idempotent to handle retries safely. Finally, the system should be containerized with Kubernetes for orchestration, using separate deployments for crawlers, API servers, and the search index, each with independent autoscaling policies based on queue depth (crawlers) and request latency (search).
